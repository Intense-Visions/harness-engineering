# ADR-001: Framework Evolution Roadmap — From Good Tool to Viral Framework

**Date:** 2026-03-22
**Status:** Accepted
**Deciders:** cwarner (framework author)

## Context

A comprehensive gap analysis of the harness engineering framework identified 112 gaps across 12 categories, including 34 game-changing capabilities that would push into uncharted territory for agentic engineering. The framework has strong foundations — 7 packages (~33,500 LOC), 49 skills, 12 personas, 37 MCP tools, a knowledge graph (12,915 nodes, 25,765 edges), and mechanical constraint enforcement — but lacks the synthesis layer that turns detection into understanding, and the community infrastructure that enables viral adoption.

The overarching goal is to maximize adoption and usability so the framework takes off virally. This ADR establishes the prioritized roadmap for addressing gaps, organized into three tiers executed sequentially.

## Decision

Execute a three-tier roadmap: **Tier 1 (quick wins)** ships in days, builds momentum and proves capability. **Tier 2 (monthly)** builds the features that create the "wow" factor and viral flywheel. **Tier 3 (quarterly)** delivers the deep intelligence that makes the framework irreplaceable.

These items require minimal new code because the infrastructure already exists. They demonstrate the framework's graph intelligence and unlock immediate value.

| 1.2      | **Automatic task independence detection**        | F5     | Pairwise file-overlap + import-chain + call-graph reachability analysis using existing ContextQL + `get_impact()`. New MCP tool: `check_task_independence`. | 1-2 days   |
| 1.3      | **Conflict prediction before parallel dispatch** | F9     | Extension of 1.2 — returns conflict matrix with reasoning instead of pass/fail. Integrate into `harness-parallel-agents` skill.                             | 1 day      |
| 1.4      | **Natural language graph queries**               | K7     | English → ContextQL parameter translation via Claude API. New MCP tool: `ask_graph`. Enables conversational codebase exploration.                           | 2-3 days   |
| 1.5      | **Test-driven architecture enforcement**         | L3     | Assertion library wrapping existing entropy/coupling/complexity metrics. Compare against baselines. Fail CI on architectural regression.                    | 3-5 days   |
| 1.6      | **Pre-commit impact preview**                    | J6     | Formatted summary from existing `get_impact()` tool. Shows "this change affects N files, M tests, K docs" before commit.                                    | 1-2 days   |

## Alternatives Considered

### Alternative 1: Feature-First (Option B from initial proposal)

Fill all missing skill domains and platform gaps before building community infrastructure.

**Why not chosen:** Missing skills (deployment, database, API design) are incremental improvements that don't create viral dynamics. Community marketplace (H1) lets the community build these skills faster than one team can. Network effects > feature completeness.

### Alternative 2: Community-First (Option C from initial proposal)

Build Discord, marketplace, and sharing before any new technical capabilities.

**Why not chosen:** Community needs something to be excited about. The Tier 1 quick wins (graph anomaly detection, natural language queries, architecture assertions) provide the "wow" content that seeds community engagement. Build the "wow," then build the community around it.

### Alternative 3: Everything in Parallel

Attempt all 112 gaps simultaneously.

**Why not chosen:** Spreads effort too thin. The tiered approach ensures each layer compounds: Tier 1 proves capability → Tier 2 creates viral flywheel → Tier 3 makes the framework irreplaceable. Each tier builds on the infrastructure of the previous one.

## Consequences

### Positive

- Tier 1 ships in 1-2 weeks, demonstrating the framework's graph intelligence capability to the community immediately
- Marketplace (Tier 2) creates network effects — each community skill makes the framework more valuable for everyone
- Game-changing capabilities (self-improving skills, spec-to-code verification, cross-project federation) move into territory no competitor occupies
- Architecture decay timeline and debt quantification convert abstract "code quality" into business-language ROI
- Natural language graph queries make the knowledge graph accessible to non-expert users
- Tiered execution means early tiers fund later tiers with adoption and feedback

### Negative

- Tier 3 items are 6-12 weeks each — long development cycles with risk of scope creep — **mitigation:** strict phase gating, ship MVPs, iterate based on usage data
- Self-improving skills (D4/D5) require outcome data that takes weeks to accumulate — **mitigation:** synthetic benchmarks initially, real data over time
- Cross-project federation (D2) adds architectural complexity (graph merging, ID scoping) — **mitigation:** opt-in, start with learnings federation only
- Multi-language support (B1) is a large surface area — **mitigation:** start with Python (largest non-JS audience), add languages driven by demand
- Marketplace requires critical mass to be useful — **mitigation:** seed with 20+ first-party skills, incentivize community contributions

### Neutral

- Some Tier 2/3 items may be reordered based on community demand signals from marketplace and Discord
- Supporting work (onboarding, testing, platform) runs continuously and may absorb sprint capacity
- The framework's competitive moat shifts from "mechanical enforcement" (defensible but niche) to "intelligence synthesis" (broader but requires sustained investment)
