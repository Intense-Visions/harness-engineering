# ADR-001: Unified Knowledge Graph for Context Assembly

**Date:** 2026-03-18
**Status:** Accepted
**Deciders:** cwarner

## Context

Harness Engineering's current context system is a collection of standalone static analyzers with no unified data model:

- **Context assembly** (`packages/core/src/context/`): Glob-based file grouping, fixed token budget ratios, hardcoded phase-to-category mappings. No semantic understanding, no relationship awareness.
- **State management** (`packages/core/src/state/`): Flat JSON + append-only markdown logs. Decisions, learnings, and failures are stored but never linked to the code they affect. No querying, no indexing.
- **Entropy detection** (`packages/core/src/entropy/`): Point-in-time snapshots rebuilt from scratch each run. File-to-file dependency edges with no semantic meaning. Dead code detection via reachability from entry points only.
- **Constraint enforcement** (`packages/core/src/constraints/`): Glob-pattern layer assignment. Import edges without weights, reasons, or cross-cutting concern awareness.
- **MCP resources** (`packages/mcp-server/src/resources/`): Five independent blobs (project, skills, rules, learnings, state) with no relationships between them.

These systems cannot answer questions like:

- "What code does this learning apply to?"
- "What's the full dependency chain from this entry point to this symbol?"
- "What decisions affected this module?"
- "Show me all files semantically related to authentication"
- "Which learnings are relevant to the code I'm about to change?"

Meanwhile, the OmniContext project (`/Users/cwarner/Projects/omnicontext`) demonstrated that a graph-based approach with 20 node types, 12 edge types, ContextQL noise-pruning, and Fusion Layer (keyword + semantic) search can achieve 99%+ context reduction while maintaining relationship awareness. OmniContext has completed Phases 1-6 (18/18 milestones) but exists as a separate project with significant overlap.

The goal is to build the best possible context system for AI agent interaction — one that understands code structure, organizational knowledge, runtime behavior, and the relationships between all of them — and deprecate OmniContext.

## Decision

Build a **Unified Knowledge Graph** as a new `packages/graph` package within the Harness Engineering monorepo. This graph becomes the single source of truth for all context assembly, replacing the existing context, entropy, constraint, and state systems.

### Core Architecture

**1. Graph Store** — LokiJS-backed in-memory graph database

Node types (minimum):

- **Code**: `repository`, `module`, `file`, `class`, `interface`, `function`, `method`, `variable`
- **Knowledge**: `adr`, `decision`, `learning`, `failure`, `issue`, `document`, `skill`, `conversation`
- **VCS**: `commit`, `build`, `test_result`
- **Observability**: `span`, `metric`, `log` (future)
- **Structural**: `layer`, `pattern`, `constraint`, `violation`

Edge types (minimum):

- **Code relationships**: `contains`, `imports`, `calls`, `implements`, `inherits`, `references`
- **Knowledge relationships**: `applies_to`, `caused_by`, `resolved_by`, `documents`, `violates`, `specifies`, `decided`
- **VCS relationships**: `co_changes_with`, `triggered_by`, `failed_in`
- **Execution relationships**: `executed_by`, `measured_by` (future)

Nodes carry optional `embedding?: number[]` for vector search.

**2. Graph Storage & Lifecycle**

The graph is a **derived artifact** — not stored in source control. It is generated from source files, documentation, state, and external systems on demand.

- **Storage location**: `.harness/graph/` (LokiJS JSON file + HNSWLib vector index + connector sync metadata)
- **`.gitignore`d**: Like `node_modules` or `dist/`, the graph is always regenerable
- **Population**: `harness scan` performs full ingestion; `harness watch` keeps it fresh via filesystem events
- **Portability**: Any developer can rebuild the graph from scratch. External connector data is cached locally but re-pullable.

**3. Ingestion Pipeline** — Multi-source graph population

The ingestion pipeline uses a **connector architecture** — pluggable adapters that pull from both local and external sources, creating graph nodes linked to code via the Fusion Layer.

#### Local Connectors (always available)

- **Code ingestion**: Tree-sitter AST parsing (TypeScript, Go, Rust, Java) with worker pool architecture. Produces code nodes + structural edges. Topological post-processing establishes cross-module relationships.
- **Knowledge ingestion**: Parses ADRs, AGENTS.md sections, learnings, failures, decisions. Links to code nodes via Fusion Layer (keyword heuristic + vector similarity).
- **State ingestion**: Converts harness state (progress, blockers, handoffs) into graph nodes linked to relevant code and knowledge nodes.
- **Constraint ingestion**: Layer definitions and forbidden-import rules become `constraint` nodes with `violates` edges to offending code.
- **Git history ingestion**: Commit messages and PR descriptions become `commit` nodes. Co-change analysis produces `co_changes_with` edges between frequently co-modified files. Blame data enriches code nodes with ownership metadata.

#### External Connectors (configured per-project)

Each external connector implements a common interface:

```typescript
interface GraphConnector {
  name: string;
  source: string;
  ingest(store: GraphStore, config: ConnectorConfig): Promise<IngestResult>;
  schedule?: string; // cron expression for auto-refresh
}
```

- **Jira / Linear / GitHub Issues**: Pulls issues via API (filtered by project, label, status). Creates `issue` nodes with metadata (assignee, status, priority, labels). Fusion Layer links issues to code nodes via keyword matching on component names + vector similarity on issue description vs code docstrings. Edges: `issue --applies_to--> function:handleAuth`, `issue --caused_by--> commit:abc123`.

- **Slack / Discord**: Ingests channel history via API (filtered by channel, timeframe, keyword). Creates `conversation` nodes with metadata (author, channel, timestamp). Named entity recognition extracts file/function/module mentions from message text. Edges: `conversation --references--> file:src/billing.ts`, `conversation --decided--> adr:003`.

- **Confluence / Notion / Google Docs**: Pulls pages via API or reads exported markdown. Creates `document` nodes with sections as child nodes. Fusion Layer links sections to code they describe. Edges: `document --documents--> module:auth`, `document_section --specifies--> interface:UserAPI`.

- **CI/CD (GitHub Actions, etc.)**: Pulls build/test results. Creates `build` and `test_result` nodes. Links failures to code nodes that caused them. Edges: `test_result --failed_in--> function:validateInput`, `build --triggered_by--> commit:def456`.

#### Connector Configuration

Connectors are configured in `harness.config.json`:

```json
{
  "graph": {
    "connectors": [
      {
        "source": "jira",
        "config": {
          "apiKeyEnv": "JIRA_API_KEY",
          "baseUrlEnv": "JIRA_BASE_URL",
          "project": "ENG",
          "filters": { "status": ["In Progress", "Done"], "labels": ["backend"] },
          "schedule": "0 */4 * * *"
        }
      },
      {
        "source": "slack",
        "config": {
          "apiKeyEnv": "SLACK_BOT_TOKEN",
          "channels": ["#engineering", "#incidents"],
          "lookbackDays": 90,
          "schedule": "0 */6 * * *"
        }
      }
    ]
  }
}
```

API keys are always referenced via environment variable names (`apiKeyEnv`), never stored as values in the config file. See [Security & Privacy](#security--privacy) for details.

#### Connector Lifecycle

- **Initial sync**: `harness ingest --source jira` (or `--all` for everything)
- **Incremental sync**: Each connector tracks `lastSyncTimestamp` in `.harness/graph/sync-metadata.json`. Only pulls new/changed items on subsequent runs.
- **Scheduled sync**: Optional cron-based auto-refresh via `schedule` field.
- **Manual refresh**: `harness ingest --source slack --full` forces a complete re-pull.
- **Linking**: After ingestion, the Fusion Layer runs a linking pass — matching new external nodes to existing code nodes via keyword + semantic similarity. Confidence scores are stored on edges.

#### How External Nodes Link to Code

The Fusion Layer uses a two-signal approach for every external node:

1. **Keyword signal (deterministic)**: Extract component names, file paths, function names, and module names mentioned in the external content. Direct string match against graph node names/paths.
2. **Semantic signal (probabilistic)**: Embed the external content (issue description, Slack message, doc section) and find top-N most similar code nodes by cosine similarity.
3. **Score fusion**: Weighted combination (default 60% keyword / 40% semantic). Edges are created only above a configurable confidence threshold (default 0.6).

This ensures external knowledge is always anchored to specific code — not floating as disconnected context.

**4. ContextQL Query Engine** — Deterministic graph traversal with noise pruning

- BFS traversal from root nodes with configurable depth, type filters, and edge filters
- Automatic noise pruning: structural queries exclude observability nodes unless explicitly requested
- DSL support for declarative queries
- Projection (field selection) to minimize context payload
- Bidirectional traversal (inbound + outbound edges)

**5. Vector Store** — HNSWLib with pure-TS fallback

- Dimension-agnostic (adapts to configured embedding model — 1536 for OpenAI ada-002, 1024 for some local models, etc.)
- Lazy loading of native binary; graceful degradation to brute-force cosine similarity in pure TypeScript
- Incremental index updates on graph mutations
- Semantic search across all node types
- Provider-tagged vectors for safe model switching (see [Embedding Provider Strategy](#embedding-provider-strategy))

**6. Fusion Layer** — Combined keyword + semantic context resolution

- Keyword extraction from natural language queries
- Vector similarity search against embedded nodes
- Score fusion: weighted combination of deterministic (keyword match) and probabilistic (cosine similarity) signals
- Intent classification pipeline: classify → discover targets → extract constraints

**7. MCP Integration** — Graph-powered tools replacing static resources

New MCP tools (see [Package Impact: mcp-server](#new-tools-6) for full details):

- `query_graph`, `search_similar`, `find_context_for`, `get_relationships`, `get_impact`, `ingest_source`

Existing MCP resources (`harness://project`, `harness://skills`, etc.) become graph queries rather than static file reads. See [Package Impact: Modified Resources](#modified-resources-4-of-5) for details.

**8. Context Assembly** — Graph-driven replacement for budget/filter/generate

- Token budgeting informed by graph density (modules with many edges get more tokens)
- Phase-aware filtering via graph queries (not glob patterns)
- AGENTS.md generation from graph structure (not glob grouping)
- Dynamic context expansion: start from intent, traverse to relevant context, prune noise

### Dependencies

**Required:**

- `lokijs` — In-memory graph storage (~1.5MB)
- `tree-sitter` + language parsers — AST parsing (~2-3MB for TS/Go/Rust/Java)
- `zod` — Schema validation (already in monorepo)

**Optional (graceful degradation):**

- `hnswlib-node` — Native vector search. Falls back to brute-force cosine similarity.
- `@ai-sdk/*` — Embedding generation. Falls back to no-embedding mode (keyword-only search).

### Migration Path

1. **Phase 1 — Graph Foundation**: Build `packages/graph` with store, query engine, and code ingestion. Validate with tests.
2. **Phase 2 — Knowledge Layer**: Add knowledge, state, and git ingestion. Wire up MCP tools (`query_graph`, `search_similar`).
3. **Phase 3 — External Connectors**: Build connector architecture + first external connectors (Jira, Slack). Incremental sync + scheduling.
4. **Phase 4 — Context Assembly**: Replace `packages/core/src/context/*` with graph-driven assembly. Create harness-context-assembler skill. Deprecate old context APIs.
5. **Phase 5 — Entropy Migration**: Replace entropy snapshot with graph queries. Dead code = unreachable nodes. Drift = stale edges. Update detect-doc-drift and cleanup-dead-code skills.
6. **Phase 6 — Constraint Migration**: Replace constraint checking with graph traversal. Layer violations = edge type violations. Update enforce-architecture skill.
7. **Phase 7 — Tier-1 Skill Migration**: Update remaining Tier-1 skills (code-review, validate-context, parallel-agents, onboarding) to use graph queries.
8. **Phase 8 — New Skills & Personas**: Create 5 new skills (impact-analysis, dependency-health, hotspot-detector, test-advisor, knowledge-mapper). Create 2 new personas (Graph Maintainer, Codebase Health Analyst). Update existing personas.
9. **Phase 9 — Tier-2 Skills & Remaining Connectors**: Update Tier-2 skills (execution, planning, architecture-advisor, refactoring, align-documentation). Add remaining connectors (Confluence, CI).
10. **Phase 10 — Deprecation**: Deprecate OmniContext. Remove old context/entropy/constraint systems. Harness graph is the single system.

Each phase is independently shippable.

### New Skills Enabled by the Graph

These skills are impossible without a relationship model. They represent capabilities that don't exist in Harness today.

**1. harness-impact-analysis**

- **Purpose**: Answer "if I change X, what breaks?"
- **How**: Reverse-traverse `calls`, `imports`, `implements` edges from changed nodes. Return affected modules, tests, documentation, and downstream consumers.
- **Cognitive mode**: analytical-reporter
- **Triggers**: on_pr, manual
- **Why it doesn't exist today**: Current system has no reverse-dependency traversal. Grep finds direct importers but misses transitive consumers.

**2. harness-dependency-health**

- **Purpose**: Analyze structural health of the codebase and surface problems before they become incidents.
- **How**: Graph metrics — hub detection (nodes with high fan-in), orphan detection (unreachable nodes), cycle detection (circular edge paths), deep chain detection (fragile dependency paths >N hops). Output: health score + specific recommendations.
- **Cognitive mode**: analytical-reporter
- **Triggers**: scheduled weekly, manual
- **Why it doesn't exist today**: Current `check-deps` only validates layer rules. No holistic health analysis.

**3. harness-hotspot-detector**

- **Purpose**: Identify modules that represent structural risk via co-change analysis.
- **How**: Combine graph structure with git history (`co_changes_with` edges). Detect: files that always change together but live apart (should be co-located), modules that change most frequently (highest risk), high logical coupling with low structural coupling (hidden dependencies).
- **Cognitive mode**: analytical-reporter
- **Triggers**: scheduled weekly, manual
- **Why it doesn't exist today**: No co-change data. No way to correlate git history with code structure.

**4. harness-test-advisor**

- **Purpose**: Answer "I changed these files — what tests should I run?"
- **How**: Traverse `imports`/`calls` edges from changed file nodes. Find all test file nodes with edges to affected nodes. Rank by graph distance (direct importers first, transitive second).
- **Cognitive mode**: advisory-guide
- **Triggers**: on_pr, manual
- **Why it doesn't exist today**: Test selection is currently "run all" or manual. No way to compute affected test surface.

**5. harness-knowledge-mapper**

- **Purpose**: Replace manual AGENTS.md maintenance with auto-generated, always-current knowledge maps.
- **How**: Query graph for module structure, entry points, dependency diagrams, API surfaces. Generate markdown documentation derived from graph — never stale because it's computed, not authored. Detect undocumented modules automatically (code nodes with no `documents` edges).
- **Cognitive mode**: constructive-architect
- **Triggers**: on_commit (main branch), scheduled daily, manual
- **Why it doesn't exist today**: AGENTS.md is manually maintained. `validate-context-engineering` checks it but can't generate it intelligently.

**6. harness-context-assembler**

- **Purpose**: The core skill for graph-powered context assembly. Takes an agent's intent and returns optimal context within token budget.
- **How**: Accept intent ("implement feature X", "fix bug Y", "review PR Z"). Use Fusion Layer to identify relevant graph nodes. Apply ContextQL noise pruning. Respect token budget via graph-density-aware allocation. Return assembled context with relationship metadata.
- **Cognitive mode**: meticulous-implementer
- **Triggers**: automatic (invoked by other skills before they begin work)
- **Why it doesn't exist today**: Context assembly is currently static (phase → glob patterns → fixed budget ratios). No intent-awareness, no relationship-awareness.

### New Personas Enabled by the Graph

**1. Graph Maintainer**

- **Role**: Keep the knowledge graph fresh, monitor connector health, detect graph anomalies, ensure data quality.
- **Skills**: harness-dependency-health, harness-knowledge-mapper, validate-context-engineering
- **Triggers**: scheduled daily (full re-scan + incremental connector sync), on_commit to main (re-ingest changed files), on connector failure (alert + retry)
- **Why**: The graph is now infrastructure — it needs a caretaker. Stale graph = stale context = agent mistakes.

**2. Codebase Health Analyst**

- **Role**: Proactively identify structural problems, coupling risks, and architectural drift before they become incidents.
- **Skills**: harness-hotspot-detector, harness-dependency-health, harness-impact-analysis, cleanup-dead-code
- **Triggers**: scheduled weekly, on_pr (for PRs touching >10 files)
- **Why**: No existing persona does proactive structural analysis. Current personas are reactive (enforce rules, fix drift, clean entropy). This persona looks forward.

#### Backward Compatibility Strategy

All modified functions support a **hybrid mode** during migration:

## Design Considerations

### Testing Strategy

**Unit tests**: Each module in `packages/graph` (GraphStore, ContextQL, VectorStore, each ingestor, each connector, FusionLayer) gets its own test suite with in-memory fixtures. No external dependencies required for unit tests.

**Connector mocking**: External connectors (Jira, Slack, Confluence, CI) are tested against recorded API response fixtures, not live services. Each connector implements `GraphConnector` interface — tests inject a mock HTTP client that returns fixture data. The `SyncManager` is tested with an in-memory clock for schedule advancement.

**Integration tests**: End-to-end tests that run `harness scan` on a fixture project (small TypeScript repo checked into `packages/graph/__fixtures__/`), then execute ContextQL queries and verify results. These validate the full pipeline: parse → ingest → link → query.

**Performance benchmarks**: Tracked in CI via Vitest bench. Key thresholds:

- `harness scan` on 10K-file repo: < 30 seconds
- ContextQL query (depth 3, 50K nodes): < 50ms
- Vector search (10K embedded nodes): < 100ms
- LokiJS memory ceiling: ~500K nodes before considering ArangoDB migration path

**Graph corruption recovery**: If `.harness/graph/` is corrupted or schema-incompatible, `harness scan` detects this on load (schema version check) and rebuilds from scratch. The graph is always regenerable — corruption is an inconvenience (re-scan), not data loss.

### Concurrency & Multi-Agent

**Single-process model**: LokiJS is single-process, in-memory. This is acceptable because:

- `harness scan` and `harness watch` are developer-machine processes, not server processes
- Parallel agents in worktrees each get their own `.harness/graph/` copy (worktrees are filesystem-isolated)
- MCP server runs in a single process and serializes graph access naturally

**Worktree isolation**: When agents use git worktrees, each worktree has its own `.harness/graph/` directory. The graph is cheap to rebuild (~seconds for typical repos), so each worktree runs its own `harness scan`. No shared graph, no locking needed.

**Future scaling**: If a shared graph becomes necessary (e.g., CI server serving multiple agents), the `GraphStore` interface abstracts over the storage backend. LokiJS can be swapped for ArangoDB (networked, multi-client) without changing query logic. This is a future optimization, not a launch requirement.

### Schema Evolution

**Version field**: The graph persistence file includes a `schemaVersion` number. On load, if the version doesn't match the current code's expected version, the graph is discarded and rebuilt via `harness scan`.

**Why rebuild-over-migrate**: The graph is a derived artifact. Unlike a database with user data, nothing in the graph is irreplaceable — it's all derivable from source files, docs, state, and external connectors. Migration scripts add complexity for zero benefit. Re-scan is the migration strategy.

**Connector data**: External connector data (Jira issues, Slack messages) is cached in the graph but re-pullable. After a schema change and re-scan, connectors re-sync. `lastSyncTimestamp` is reset, triggering a full re-pull.

### Embedding Provider Strategy

**Provider-tagged vectors**: Each embedding stored in the vector index includes a `provider` + `model` tag in its metadata:

```typescript
{
  nodeId: "function:handleAuth",
  embedding: [0.12, -0.34, ...],
  provider: "openai",
  model: "text-embedding-ada-002",
  dimensions: 1536
}
```

**Provider switch = re-embed**: If the configured embedding provider changes, the vector index is invalidated and rebuilt. This is triggered automatically when the provider/model in config doesn't match the provider/model in the index metadata.

**Graceful degradation**: If no embedding provider is configured (no API key), the vector store operates in **keyword-only mode**. The Fusion Layer falls back to 100% keyword signal, 0% semantic signal. This means the graph works fully without any AI API — embeddings are an enhancement, not a requirement.

**Dimensionality**: The vector store is dimension-agnostic. HNSWLib is initialized with whatever dimensions the configured model produces (1536 for ada-002, 1024 for some local models, etc.). Switching dimensions requires re-indexing.

### Security & Privacy

**API key storage**: Connector API keys are stored in environment variables, NOT in `harness.config.json`. The config file references variable names:

```json
{
  "graph": {
    "connectors": [
      {
        "source": "jira",
        "config": {
          "apiKeyEnv": "JIRA_API_KEY",
          "baseUrlEnv": "JIRA_BASE_URL"
        }
      }
    ]
  }
}
```

**Sensitive data in graph**: External connector data (Slack messages, Jira descriptions) may contain PII or secrets. Mitigations:

- The graph is `.gitignore`d — never committed to source control
- The graph is local to each developer's machine — no centralized storage
- Connectors support content filtering (e.g., only ingest issue titles + labels, not full descriptions)
- Optional redaction patterns in connector config: `"redact": ["\\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\\.[A-Z|a-z]{2,}\\b"]` strips emails before ingestion

**Embedding privacy**: When using external embedding APIs (OpenAI, Anthropic), code content is sent to the provider for embedding. For sensitive codebases, configure a local embedding model (Ollama) to keep all data on-machine.

### Developer Experience & Onboarding

**Graph is optional**: All existing commands work without a graph. The hybrid mode (graph-first, config-fallback) means:

- No graph → commands behave exactly as they do today
- Graph exists → commands use graph for richer results
- Graph is stale → commands still work, with a warning

**First-run experience**:

```bash
harness init          # scaffolds project (unchanged)
harness scan          # NEW: builds graph (optional, ~5-15s for typical repos)
harness validate      # works with or without graph
```

**Staleness detection**: Every command that reads the graph checks the `lastScanTimestamp` in `.harness/graph/metadata.json`. If the graph is older than a configurable threshold (default: 24 hours), a warning is emitted:

```
⚠ Graph is 3 days old. Run `harness scan` to refresh.
```

Commands never fail due to staleness — they use stale data with a warning.

**Tree-sitter portability**: Tree-sitter requires native compilation, which can fail on some platforms. Mitigations:

- Tree-sitter is an optional dependency. If it fails to install, `harness scan` falls back to regex-based parsing (extracts imports/exports but not full AST). This produces a less detailed graph but still functional.
- Pre-built binaries are available for common platforms (macOS ARM/x86, Linux x86, Windows x86). The `optionalDependencies` pattern in package.json handles this.
- CI environments can use the Docker image which includes pre-compiled tree-sitter.

### Offline / Disconnected Mode

The graph operates in two layers:

1. **Local layer** (always available): Code parsing, doc ingestion, state ingestion, git history, constraint ingestion. These work entirely offline.
2. **External layer** (requires network): Jira, Slack, Confluence, CI connectors. These require API access.

When offline:

- Local layer is fully functional. `harness scan` rebuilds code/doc/state graph.
- External layer uses cached data from last successful sync. Connector nodes remain in the graph with their last-known state.
- `harness ingest` commands fail gracefully with "connector unavailable — using cached data" message.
- No graph queries fail. External nodes may be stale but are still queryable.

Embedding generation also requires network (for cloud providers). When offline with no local model configured, the vector store operates in keyword-only mode.

## Alternatives Considered

### Graph Overlay (Option B)

Add a graph layer alongside existing systems via adapters. Existing context/entropy/constraints continue as-is. Graph observes their outputs and provides semantic search and relationship queries on top.

Not chosen because: Creates permanent dual-system complexity. The graph is only as good as what existing systems expose — missing AST-level relationships. The "overlay" pattern tends to become permanent tech debt. Given no hard constraints on timeline or breaking changes, a clean replacement is preferable.

### Context Query Engine (Option C)

Build an on-demand query engine that constructs graph-like views at query time from source files. No persistent graph. Vector index maintained separately.

Not chosen because: Performance degrades at scale (parsing files on every query). Cannot answer temporal or cross-cutting queries without full scans. Caching complexity risks creating an accidental graph store. Limited capability ceiling doesn't match the goal of "best possible context system."

## Consequences

### Positive

- **Unified data model**: One graph replaces 6 disconnected systems (context, entropy, constraints, state, MCP resources, dependency tracking)
- **Semantic search**: Find similar code, related learnings, applicable decisions via vector similarity — not just string matching
- **Relationship-aware context**: Agents receive context that includes not just files but the connections between them
- **99%+ context reduction**: ContextQL noise pruning delivers only relevant nodes, not entire file trees
- **Extensible**: New node types (telemetry, CI results, PR reviews) can be added without schema changes
- **OmniContext deprecated cleanly**: Single system, no overlap, no confusion

### Negative

- **3-4 month build** — mitigated by phased delivery (each phase independently shippable) and existing domain knowledge from OmniContext
- **Tree-sitter native dependency** — mitigated by lazy loading, optional installation, and regex-based fallback for basic parsing
- **Breaking changes to existing context APIs** — mitigated by adapter layer during transition and phased migration
- **Embedding cost** — mitigated by optional embedding generation, local model support (Ollama), and keyword-only fallback
- **Operational surface area** — mitigated by LokiJS simplicity (single JSON file) and no external database requirements

### Neutral

- Existing skills that use `harness://project` and other MCP resources will need updating to use graph-powered equivalents, but the API surface can remain similar
- The `packages/core/src/entropy/` snapshot model has value as a "build once, query many" pattern that the graph adopts (scan → persist → query)
