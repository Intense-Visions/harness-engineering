# Plan: roadmap-shard-store Phase 6 — Rollout

**Date:** 2026-06-28 | **Spec:** docs/changes/roadmap-shard-store/proposal.md (Phase 6) | **Tasks:** 19 | **Time:** ~80 min | **Integration Tier:** large

## Goal

Make roadmap sharding discoverable, documented, and the default for new projects — sharded mode formally recognized by `load-mode.ts`, `harness init` scaffolds a sharded roadmap, and the rationale (read-source invariant, slug↔issue identity, auto-done, hook/merge-driver mechanics) is captured in ADRs, knowledge entries, AGENTS.md, and the skills — while existing adopters keep working unchanged and opt in via `harness roadmap shard`.

## Observable Truths (Acceptance Criteria)

1. The system shall expose a single sharded-storage detector in `packages/core/src/roadmap/load-mode.ts` (barrel-exported), and `store/factory.ts` shall use that one detector instead of its own inline `exists(shardDir)` check — no two detection mechanisms that can disagree.
2. When `docs/roadmap.d/` exists, the detector shall report `sharded`; otherwise `monolith`. All current `RoadmapMode` consumers shall continue to compile and behave unchanged (no unhandled-enum breakage).
3. When `harness init` scaffolds a **new** project, it shall create `docs/roadmap.d/_meta.md` (empty milestone list, project frontmatter) and shall NOT create a monolith `docs/roadmap.md`. When run against an existing project, it shall not create or migrate any roadmap file.
4. ADR `0050` (read-source invariant R) and ADR `0051` (D2: slug identity + External-ID durable sync key) shall exist under `docs/knowledge/decisions/`, following the existing ADR format.
5. Knowledge entries for the roadmap-store abstraction, shard layout, read-source invariant, regeneration determinism, slug↔issue identity, and merge-triggered auto-done shall exist under `docs/knowledge/roadmap/`.
6. The AGENTS.md Project Roadmap section shall document sharded mode, invariant R, auto-done on merge, the husky regen hooks, and the `merge.ours.driver` requirement for existing clones.
7. `harness-brainstorming` SKILL.md steps 7–8 shall stage the written shard **and** the regenerated aggregate; `harness-execution`, `harness-roadmap`, `harness-roadmap-pilot`, and `harness-onboarding` shall reflect sharded mode and auto-done (agents stop hand-marking rows).
8. An adoption/rollout guide shall document the husky mechanism, the per-clone `merge.ours.driver` step, the C3 `_meta.md` contention limit, and the offline-reconcile `state_reason` follow-up.
9. The C7 decision (`overrides.roadmapPath` drop vs display-only) shall be recorded and applied.
10. `harness validate` shall remain at baseline (365, zero NEW), and the invariant-R guard (`roadmap-read-source.repo.test.ts`) shall stay green (no new `roadmap.md` content readers introduced by docs/code).

   ```ts
   /**
    * Roadmap storage layout — orthogonal to {@link RoadmapMode} (which selects the
    * canonical source: file-backed vs file-less). Storage layout selects how a
    * file-backed roadmap is physically stored:
    *   - `monolith` — a single `docs/roadmap.md` aggregate is canonical (legacy).
    *   - `sharded`  — per-row shards under `docs/roadmap.d/` are canonical; the
    *     aggregate is a generated `merge=ours` view (spec Decisions D3/D4/R).
    *
    * Auto-detected by the presence of `docs/roadmap.d/`. This is the single
    * detection authority; `store/factory.ts` delegates here so the formal mode and
    * the store backend can never disagree.
    *
    * @see docs/changes/roadmap-shard-store/proposal.md (§Mode detection)
    */
   export type RoadmapStorageMode = 'monolith' | 'sharded';

   export function detectRoadmapStorageMode(
     projectRoot: string,
     exists: (target: string) => boolean = (target) => fs.existsSync(target)
   ): RoadmapStorageMode {
     return exists(path.join(projectRoot, 'docs', 'roadmap.d')) ? 'sharded' : 'monolith';
   }
   ```

   (Add `import * as path from 'node:path';` is already present; `fs` is already imported.)
