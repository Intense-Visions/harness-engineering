# Shard the roadmap into per-row files with a generated aggregate view

## Overview & Goals

### Problem

`docs/roadmap.md` is a single ~109 KB serialized file (170 rows across 24 milestones)
rewritten in full by every status change, promotion, assignment, groom, and inbound
sync. Because each write re-serializes the whole file via `serializeRoadmap`
(`packages/core/src/roadmap/serialize.ts`), logically independent row edits collide at
the text level — producing **187 commits / 26 merge-conflict resolutions in the last 90
days** plus recurring pre-push `format:check` failures on the live-drifting file.
`serializeRoadmap` is also lossy (drops prose/comments), so naive row-level merges
cannot safely round-trip.

A second, related pain: a row is only marked `done` **after** its implementing PR
merges, but there is no mechanism to do this — it is a manual cleanup commit every time
(e.g. `chore(roadmap): mark #528 and #640 done — completed work was stale as planned`).
Rows therefore drift stale against shipped code. The reason auto-done was never built is
the same contention problem: a bot rewriting the 109 KB monolith on every merge would
collide with every concurrent edit. Sharding removes that blocker.

Every one of the 170 rows already mirrors to a GitHub issue (`External-ID:
github:...#NNN`, 170/170), and `packages/core/src/roadmap/sync-engine.ts` already
performs bidirectional push/pull. `.gitattributes` already uses `merge=ours` for
auto-generated baseline/timeline files — the exact escape hatch — but excludes
`roadmap.md` because it currently carries human-authored content. The archive split
(`docs/roadmap-archive.md`) is precedent for breaking the file apart.

### Goal

Eliminate roadmap contention **by construction** — disjoint row edits must never touch
the same file — while keeping the roadmap git-native, offline-capable, human-browsable,
and PR-reviewable, and without coupling correctness to the (flaky) MCP↔GitHub sync path.
Then use the conflict-free writeback that sharding unlocks to **automatically mark a row
`done` when its implementing PR merges**.

### Approach (locked via brainstorming)

Per-row shard files under `docs/roadmap.d/<slug>.md` become the **sole authoritative
source**. `docs/roadmap.md` becomes a **generated, `merge=ours` aggregate view**
regenerated by a deterministic hook. The existing GitHub sync-engine stays as the
optional cross-machine reconciliation layer. A `RoadmapStore` abstraction lets sharded
and monolith modes coexist so adopters migrate deliberately. A merge-triggered
reconciler flips the corresponding shard to `done` when a PR closes its linked issue.

### Goals

1. Two actors editing different rows never produce a merge conflict (disjoint files).
2. The generated `roadmap.md` never causes a code path to be wrong (read-source
   invariant R).
3. Existing adopters keep working unchanged until they opt in; new projects are sharded
   by default.
4. Eliminate the pre-push `format:check` live-drift failure via a single deterministic
   regenerator.
5. A row reaches `done` automatically when its implementing PR merges — no manual
   cleanup commit.

### Non-goals (YAGNI)

- Not replacing GitHub issues as the cross-machine reconciliation layer (kept as-is).
- Not auto-migrating existing adopters' files (opt-in only).
- Not redesigning the `Roadmap` in-memory type, the milestones model, or sync adapters.
- Not supporting trackers beyond the existing pluggable adapter set.
- Not auto-advancing any status other than `→ done` on merge (no auto-promote, no
  auto-in-progress beyond what execution already claims).

## Decisions Made

| #      | Decision                                                                                                                | Rationale                                                                                                                            |
| ------ | ----------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **D1** | Hybrid: per-row local shards **plus** keep GitHub sync as cross-machine reconciliation                                  | Conflict-free local model without coupling correctness to the flaky MCP↔GitHub path; sync stays optional and pluggable               |
| **D2** | Per-row files named by **feature slug** (`docs/roadmap.d/<slug>.md`); GitHub issue # stored *inside* as `External-ID`   | Slug exists before the issue does (fixes the Intake/new-row case); browsable; the issue # stays the durable immutable sync key       |
| **D3** | `docs/roadmap.md` becomes a **generated `merge=ours` aggregate**; shards are the source of truth                        | One human/PR-reviewable artifact that cannot conflict; the internal `Roadmap` type is unchanged                                      |
| **D4** | **Hook-driven** regeneration; writers patch exactly one shard                                                            | A single deterministic regen path kills the live-drift `format:check`; writers stay "dumb", which is the conflict-free guarantee     |
| **D5** | New projects sharded by default; existing adopters migrate via a reversible `harness roadmap shard`                     | No surprise rewrites of adopter files or git history                                                                                 |
| **D6** | **Auto-done on merge:** a reconciler flips a shard to `done` when a merged PR closes the row's linked `External-ID` issue | Sharding makes the writeback conflict-free, removing the blocker that kept this manual; reuses the existing issue↔row mapping         |
| **R**  | **Read-source invariant:** only the regenerator reads `roadmap.md`; all tools read `docs/roadmap.d/`                    | Makes correctness independent of hooks/merge-driver setup — the worst case is a stale cosmetic file, never wrong tool behavior       |

## Technical Design

### Shard format — `docs/roadmap.d/<slug>.md`

Reuses today's row-bullet block verbatim; the milestone (previously implied by the
containing `##` heading) moves into frontmatter:

```markdown
---
slug: protect-config-fail-closed
milestone: v5.0 — Enforcement Hardening
order: 12
---

### Make protect-config fail-closed in ambiguous cases

- **Status:** planned
- **Spec:** docs/changes/protect-config-fail-closed/proposal.md
- **Summary:** ...
- **Blockers:** —
- **Plan:** —
- **Assignee:** —
- **Priority:** P1
- **External-ID:** github:Intense-Visions/harness-engineering#527
- **Updated-At:** 2026-06-27T12:59:34.263Z
```

The row body parser is unchanged; only the milestone source moves to frontmatter.

### `_meta.md` (frontmatter-only)

Holds the old roadmap frontmatter (project, version, timestamps) **plus the ordered
milestone list** — the single source of milestone ordering. Edited only when a milestone
is added or reordered (rare; a trivial append that virtually never conflicts).

### `RoadmapStore` interface (core)

Two backends behind one interface that returns the same in-memory `Roadmap`:

- **`MonolithStore`** — current `parseRoadmap`/`serializeRoadmap` on `roadmap.md`
  (behavior unchanged).
- **`ShardStore`** — globs `docs/roadmap.d/*.md`, parses each row block, groups by the
  `milestone` frontmatter, orders milestones via `_meta` then rows via
  `order`/status-rank/slug. `patchFeature(slug, mutation)` rewrites **exactly one**
  shard. `addFeature` writes a new shard.

Because `load()` returns the same `Roadmap` object in both modes, existing call sites
are unchanged.

### Deterministic regenerator

`_meta` milestone order + shards → `Roadmap` → `serializeRoadmap` → `roadmap.md`. Same
shards ⇒ byte-identical output. This determinism is what fixes the `format:check`
live-drift failure — a regenerated aggregate is always prettier-clean and stable.

### Auto-done reconciler

A reconciler maps a closed GitHub issue back to its shard via `External-ID` and patches
that shard's status to `done`, reusing the assignee-lifecycle rule (assignee cleared
when leaving `in-progress`). Two trigger surfaces, sharing one core function:

- **CI (authoritative):** a GitHub Action on `pull_request: closed` (merged) reads the
  PR's closed issues, patches each matching shard to `done`, regenerates the aggregate,
  and commits to `main`. Because each row is its own file, the commit touches only the
  shipped row(s) — conflict-free with any concurrent in-flight work.
- **Local fallback:** `harness roadmap reconcile` performs the same mapping from current
  issue state, for offline/manual catch-up. Idempotent — a row already `done` is a no-op.

A PR with no roadmap-linked closed issue is a no-op. A PR closing multiple issues patches
multiple shards (each its own file → still conflict-free).

### Mode detection

`packages/core/src/roadmap/load-mode.ts` gains a `sharded` mode, auto-detected by the
presence of `docs/roadmap.d/`.

### Blast-radius reduction

The live MCP "flips unrelated rows" defect becomes **structurally impossible**: a buggy
write (or auto-done reconcile) can only touch a single shard, never clobber the other
169 rows.

## Integration Points

### Entry Points

- New CLI commands: `harness roadmap shard`, `harness roadmap unshard`,
  `harness roadmap regen`, `harness roadmap reconcile`.
- New core modules: `RoadmapStore`, `ShardStore`, the assembler, the regenerator, and the
  reconciler.
- New git hook: pre-commit + post-merge regeneration of the aggregate.
- New CI workflow: PR-merge auto-done reconcile.

### Registrations Required

- Register the regen hook in **both** `packages/cli/.../profiles.ts` AND
  `plugin-config.mjs STANDARD_HOOKS` (the hook channel silently desyncs otherwise — no
  test catches it).
- `.gitattributes`: add `docs/roadmap.md merge=ours`.
- `harness init`: run `git config merge.ours.driver true` (already required today for
  baseline files; make it explicit).
- Register the auto-done GitHub Action workflow.
- Barrel exports for the new core modules.

### Documentation Updates

- AGENTS.md roadmap section.
- `harness-brainstorming` SKILL.md steps 7–8 (promote now writes a shard and the
  regenerated aggregate, then commits both).
- `harness-execution`, `harness-roadmap`, `roadmap-pilot` skills; note auto-done so
  agents stop hand-marking rows.
- Onboarding docs.

### Architectural Decisions

- **R (read-source invariant)** — rises to a standalone ADR: it is a load-bearing system
  invariant that is mechanically enforced and that downstream correctness depends on.
- **D2 (slug identity + `External-ID` as the durable sync key)** — rises to a standalone
  ADR: it defines cross-system identity between local shards and external trackers, and
  is the mapping the auto-done reconciler depends on.

### Knowledge Impact

Roadmap-store abstraction; shard directory layout; the read-source invariant;
regeneration determinism; slug↔issue identity mapping; merge-triggered auto-done.

## Success Criteria

- Two branches editing distinct rows merge with **zero conflicts** (integration test
  simulating two branches).
- No code path reads `roadmap.md` except the regenerator — enforced by a grep/validate
  check (invariant R).
- `harness roadmap shard` migrates this repo's 170 rows with a **semantic round-trip**:
  `parse(old)` deep-equals `parse(regen(shards))`. (Byte-equality against the *old* file
  is impossible because today's serialize is already lossy; this is no regression.)
- Pre-push `format:check` passes on a freshly regenerated aggregate, and reruns are
  byte-stable.
- `harness roadmap unshard` losslessly reverses the migration.
- Merging a PR that closes a roadmap-linked issue flips exactly that row to `done`
  (touching only its shard) with no manual commit; `harness roadmap reconcile` is
  idempotent.

## Implementation Order

1. **Core foundation** — shard format, `RoadmapStore` + `ShardStore`, assembler, and the
   deterministic regenerator; unit tests including semantic round-trip and
   byte-stability.
2. **Migration CLI** — `shard` / `unshard` / `regen` with the round-trip assertion.
3. **Git/hook integration** — regen hook (both profiles + plugin-config),
   `.gitattributes merge=ours`, `init` git-config, and the read-source invariant check.
4. **Wire writers through the store** — `manage_roadmap`, brainstorming promote,
   execution claim, pilot assign, groom, and sync-engine — each patches a single shard.
5. **Auto-done reconciler** — core reconcile function + `harness roadmap reconcile` CLI +
   PR-merge GitHub Action; idempotency and assignee-lifecycle tests.
6. **Rollout** — `sharded` mode detection, new-init scaffolds sharded, docs/ADRs/knowledge.
7. **Dogfood migration** — run `harness roadmap shard` on this repo; commit the shards
   plus the generated aggregate.
